How iverilog decides which files to compile, which one to run, and how to take control when needed.

01

Why * Breaks and *.v Fixes It

After you compile once, a file called a.out appears in the folder. The bare * wildcard grabs everything — including that compiled output. iverilog tries to read it as Verilog code and immediately crashes.

Using *.v limits the grab to only files ending in .v, so the compiled output is never touched.

Files in the folder:

mux2to1.v
design
mux4to1.v
design
tb_mux.v
test file
a.out
compiled output

With * — grabs all 4 files including a.out:

mux2to1.v
mux4to1.v
tb_mux.v
a.out
Result
a.out:1: syntax error — I give up.

With *.v — grabs only the 3 Verilog files:

mux2to1.v
mux4to1.v
tb_mux.v
a.out
Result
Compiles cleanly. a.out is skipped entirely.
02

How iverilog Picks What to Run

iverilog reads all the files, then asks: which file uses all the others but is itself used by nothing? That file is the runner — it becomes the entry point of the simulation.

In the example below, tb_mux4to1.v uses mux4to1.v, which uses mux2to1.v. But nothing uses the test file — so it gets picked automatically.

mux2to1.v
used by mux4to1
→
mux4to1.v
used by tb
→
tb_mux4to1.v
▲ picked as runner
The Rule
The file that uses others but is used by nothing becomes the runner. iverilog figures this out automatically — no extra flags needed.
03

Unrelated Files Are Harmless

If a design file exists in the folder but is not connected to anything the runner uses, iverilog compiles it but never runs it. It has zero effect on the simulation output.

In this example, adder.v is in the same folder. The test file only runs the mux — so the adder is compiled and then completely ignored.

mux2to1.v
used by mux4to1
→
mux4to1.v
used by tb
→
tb_mux4to1.v
▲ runner
╌╌
adder.v
compiled, ignored
iverilog *.v    # adder.v is compiled but has no effect on output
Safe but only to a point
This works fine as long as there is only one test file. The moment a second test file appears (like tb_adder.v), iverilog gets confused about what to run — that's when you need the force flag.
04

The Force Flag — -s

When two test files exist in the same folder, iverilog finds two potential runners and either runs both or errors. The -s flag forces it to use a specific one and ignore everything unreachable from it.

Now the project has both tb_mux4to1.v and tb_adder.v. Without -s, both run. With -s tb_adder, only the adder test runs — and all the mux files become the ignored ones.

Without -s — both test files run:

mux2to1.v
mux4to1.v
tb_mux4to1.v
▲ runs
adder.v
tb_adder.v
▲ also runs

With -s tb_adder — only adder test runs, mux files ignored:

mux2to1.v
ignored
mux4to1.v
ignored
tb_mux4to1.v
ignored
adder.v
used
tb_adder.v
▲ forced runner
iverilog -s tb_adder *.v
Module name, not filename
-s takes the name written inside the file after the word module, not the filename. Use -s tb_adder, not -s tb_adder.v.

Quick Reference

CommandWhen to use it
iverilog *.vOne test file in the folder — automatic runner detection
iverilog *.v -o sim.outSame, but give the output a real name instead of a.out
iverilog -s tb_name *.vMultiple test files — force a specific one to run
iverilog *Never — will grab a.out and crash
